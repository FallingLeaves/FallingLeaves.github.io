选项 el、propsData 的合并策略

在非生产环境下，strats的el和propsData是一个函数，来对el和propsData进行处理，该函数中参数vm来自mergeOptions函数的第三个参数vm，在_init(core/instanc/init.js)中调用了mergeOptions，vm是当前Vue实例

mergeOptions除了在_init中调用外，还在core/global-api/extend.js文件中Vue.extend中调用

此时调用mergeOptions函数就没有传递第三个参数，也就是说通过 Vue.extend创建子类的时候mergeOptions会被调用，此时策略函数就拿不到第三个参数。

策略函数中通过判断是否存在 vm 就能够得知 mergeOptions 是在实例化时调用(使用 new 操作符走 _init 方法)还是在继承时调用(Vue.extend)，而子组件的实现方式就是通过实例化子类完成的，子类又是通过 Vue.extend创造出来的，所以我们就能通过对 vm 的判断而得知是否是子组件了。

strats.el和strats.propsData的策略函数最终都返回defaultStrat(parent, child)

defaultStrat是一个默认合并策略，只要子选项不是undefined那么就是用子选项，否则使用父选项。

选项data的合并策略

在strats策略对象上添加data的合并，首先判断vm是否传递，没有vm参数时，是处理子组件的选项，判断子组件的选项是否存在并且为函数，是则返回mergeDataOrFn(parentVal, childVal)，否则返回parentVal，不是处理子组件选项时，返回mergeDataOrFn(parentVal, childVal, vm)

data的合并策略无论是否是子组件的选项，最终都是返回mergeDataOrFn函数的结果，只是参数不同

strats.data 策略函数在处理子组件的 data 选项时返回的总是一个函数，这也就间接导致 strats.data 策略函数在处理子组件选项时返回的也总是一个函数。处理new创建的实例时，返回mergedInstanceDataFn函数。

data 选项最终被处理成了一个函数，当合并处理的是子组件的选项时 data 函数可能是以下三者之一：

  • 就是 data 本身，因为子组件的 data 选项本身就是一个函数，返回childVal
  • 父类的 data 选项，返回parentVal
  • mergedDataFn函数 或 mergedInstanceDataFn函数

mergedDataFn函数和mergedInstanceDataFn函数都调用了mergeData，mergeData接受的是两个纯对象

set函数来自core/observer/index.js是Vue暴露给我们的全局API(Vue.set)

为什么最终 strats.data 会被处理成一个函数？

这是因为，通过函数返回数据对象，保证了每个组件实例都有一个唯一的数据副本，避免了组件间数据互相影响。

为什么不在合并阶段就把数据合并好，而是要等到初始化的时候再合并数据？

在合并阶段strats.data将被处理成一个函数，但是这个函数并没有被执行，而是到了后面初始化的阶段才执行的，这个时候才会调用mergeData对数据进行合并处理，那这么做的目的是什么呢？
